## What v2 does

Take a (state, year, forecast date) tuple and produce:

1. **A point estimate** of corn-for-grain yield in bushels/acre, aggregated to the state from county-level forecasts.
2. **A cone of uncertainty** (e.g. 10th–90th percentile band) derived from analog historical seasons that match the current season's weather and geospatial trajectory up to the forecast date.
3. **A natural-language forecast narrative** generated by the Claude agent that summarizes the prediction, names the analog years driving it, and explains the trajectory in plain English.

Forecasts are produced at **four fixed points in the season**:

- **August 1** — Early grain fill
- **September 1** — Late grain fill / dough stage
- **October 1** — Maturity / early harvest
- **End of Season** — Post-harvest reconciliation

For the **2025 season**, across **five states**: Iowa, Colorado, Wisconsin, Missouri, Nebraska.

## Locked decisions (do not re-litigate)

These are settled. If we reverse one, log it as a new decision in `PHASE2_DECISIONS_LOG.md`.

- **Spatial unit:** train and predict at **county level**, aggregate to state by area-weighted (planted-acres-weighted) mean. Brief explicitly suggests this; gives ~80× more training rows than state-only.
- **Forecast dates:** Aug 1, Sep 1, Oct 1, end-of-season. Locked by brief.
- **States:** IA, CO, WI, MO, NE. Locked by brief.
- **Target variable:** `yield_bu_acre_all` — combined-practice corn grain yield in bu/acre from USDA NASS QuickStats. Locked by brief.
- **Time range: 2005–2024**, per the brief's explicit "Data spans from 2005-2024" instruction. Train: 2005–2022 (18 years). Val: 2023. Holdout: 2024.
- **Foundation model:** Prithvi (NASA/IBM, HuggingFace) is the named model in the brief. Default integration mode is **frozen Prithvi as a feature extractor** (Phase D.1). End-to-end fine-tune (D.2) is a stretch goal contingent on D.1 ablation showing lift.
- **Imagery:** MODIS NDVI (via Earth Engine, server-side, masked to corn pixels via CDL) is the primary remote-sensing feature for the engineered baseline. Raw HLS is deferred to Phase D.1 because Prithvi was pretrained on it. NAIP is tertiary, used only if a phase explicitly calls for sub-meter visual context.
- **Cone-of-uncertainty methodology:** **analog-year retrieval**, not quantile regression or Bayesian intervals. Brief is explicit ("historical seasons with analogous weather patterns ... actual NASS reported outcomes from those analog years"). The cone is interpretable and tells a story; that's the point.
- **Point-estimate model:** XGBoost (or LightGBM) on engineered features as the v2 baseline. Prithvi-derived features layer on top in a later phase.
- **Cropland mask:** USDA Cropland Data Layer (CDL) is required for masking corn pixels. Currently used inside the Earth Engine NDVI script; a standalone download may be needed if the HLS pipeline is built (Phase D.1).
- **Backend:** reuse v1's FastAPI app. Add new endpoints alongside; don't break v1.
- **Frontend:** reuse v1's vanilla JS shell. Add new panels; don't migrate to React.
- **Agent role:** Claude Haiku 4.5 narrates a precomputed forecast object. Tools surface analog years, driver attributions, and historical comparisons. The agent does **not** run the regression itself.
- **Repo conventions:** v1 docs and code untouched. v2 code lives in `scripts/` (one-off pulls and feature scripts) and a new `forecast/` Python package (modeling and retrieval logic). v2 frontend additions go in `frontend/forecast/`. Phase docs all prefixed `PHASE2_`.
- **Pipeline philosophy:** for every external data source, separate the *raw pull* from the *feature engineering*. API calls and downloads are slow and rate-limited; features iterate fast. Splitting them prevents re-hitting external services on every feature change. (Already followed in `nass_pull.py` / `nass_features.py`.)
- **Join key convention:** `GEOID` (5-digit zero-padded string: state FIPS + county FIPS) and `year` (int). For per-forecast-date tables, additional key is `forecast_date` ∈ `{"08-01", "09-01", "10-01", "EOS"}`.

## Three pillars (deliberately separated)

Same architectural philosophy as v1 — keep the measurement, the model, and the narrative as separate artifacts so each is independently auditable.

1. **Measurement layer:** MODIS NDVI (corn-masked, county-aggregated) + PRISM/gridMET daily weather (GDD, EDD, VPD, precip) + gSSURGO soil features (Valu1 table aggregated to county) + NASS auxiliary (planted/harvested acres, irrigation share). Optional Prithvi embeddings on raw HLS in Phase D.1.
2. **Forecast layer:**
   - Point estimate: XGBoost regressor on the feature vector → `yield_bu_acre`.
   - Cone of uncertainty: K-nearest-neighbors over the feature vector against the historical training set → list of analog (county, year) pairs → percentile band over their observed yields.
3. **Narrative layer:** Claude Haiku 4.5 with tools that read the structured forecast and analog-year list; produces a plain-English forecast report with cited drivers and named historical analogs.

## Portfolio framing for v2

> "I built a county-level corn yield forecasting pipeline that produces calibrated uncertainty cones using historical analog seasons. The point estimate is an XGBoost regressor over weather, soil, and remote-sensing features; the uncertainty cone is K-nearest-neighbor retrieval over the same feature space, returning the percentile band of yields from the K most similar historical (county, year, date) tuples. A Claude agent narrates the forecast and names the analog years. Foundation model (Prithvi) features are layered in as an ablation study to quantify what the deep model adds over hand-engineered features."

The "interpretable analog cone + foundation-model ablation" framing is what makes this defensible in a portfolio interview. The judges (and a future interviewer) can both see exactly where the uncertainty comes from.
